Kata TDD

Déroulement de la session

  • Présentation rapide du TDD
  • Kata sur un thème choisi
  • Mise à disposition du code

Principe

C'est une méthode de conception

  1. Écrire le test (rouge)
  2. Implémenter (vert)
  3. Refactorer (vert)

Écriture du test

  • Il doit échouer
  • Il faut trouver le plus petit incrément possible
  • Le nom du test doit être explicite ; ne pas hésiter à faire des phrases
  • Il ne faut pas tester les dépendances externes
  • Les tests peuvent servir d'exemples d'utilisations

Implémentation

  • Il faut faire passer le test le plus rapidement possible
  • Il ne faut écrire que le strict nécessaire pour faire passer le test

Refactoring

  • Netoyage du code (implémentation et test)
  • On ne change que la façon de faire, pas le comportement
  • Toujours refactorer par de petits incréments

Avantages

  • Meilleur qualité du code (couplage, simplicité)
  • Sécurité face au refactoring
  • Sert de documentation sur le code
  • Feedback très rapide
  • Assure une bonne couverture de test, et minimise les régressions
  • Permet de se concentrer sur des problèmes plus simples
  • Donne confiance en du code de production

Sujets

Numération romaine

  • I (1), V (5), X (10), L (50), C (100), D (500), M (1 000)
  • Chiffres allant de 0 à 5000 exclus
  • Un même symbole ne peut pas être répeté quatre fois de suite (sauf M)
  • Un symbole qui suit un symbole de valeur supérieure ou égale s’ajoute à celui-ci
  • Un symbole qui précède un symbole de valeur supérieure se soustrait à ce dernier

Score au bowling

  • Lancer de 10 frames
  • Strike: 10 points plus le nombre de quille abbattues aux deux lancers suivants
  • Spare: 10 points plus le nombre de quille abbattues au lancer suivant
  • Trou: nombre de quilles abbatues

FizzBuzz

  • Écrire un programme affichant les nombre de 1 à 100
  • Les multiples de 3 sont remplacés par "Fizz"
  • Les multiples de 5 sont remplacés par "Buzz"
  • Les multiples de 3 et 5 sont remplacés par "FizzBuzz"
